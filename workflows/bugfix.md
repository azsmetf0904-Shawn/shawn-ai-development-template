# Workflow：Bug 修復 SOP

從 Bug 發現到修復上線的完整流程。

---

## 嚴重程度分級

| 等級 | 描述 | 處理時間 |
|------|------|----------|
| P0：緊急 | 系統無法使用、資料遺失風險、安全漏洞 | 立刻處理 |
| P1：高 | 核心功能異常，大量用戶受影響 | 當天處理 |
| P2：中 | 部分功能異常，有 workaround | 本週處理 |
| P3：低 | 顯示問題、小功能異常 | 下個迭代處理 |

---

## Step 1：記錄 Bug（2 分鐘）

把 Bug 記錄在 `TASKS.md` 的 Bugs 區塊：

```markdown
| B00X | [描述問題] | [P0/P1/P2/P3] | [日期] |
```

---

## Step 2：重現問題（5-10 分鐘）

**目的**：確保你能穩定重現問題，才能確認修好了。

**動作**：
1. 找到重現步驟（能穩定重現才算找到）
2. 確認問題的邊界：
   - 所有用戶都有，還是特定用戶？
   - 所有環境都有，還是只有 production？
   - 什麼時候開始出現的？（對應最近哪個 commit？）

**如果重現不了**：先記錄已知資訊，等用戶再次回報

---

## Step 3：找根本原因（用 AI 協助）

用 `prompts/debugging.md` 的模板：

```
1. 描述問題（預期 vs 實際行為）
2. 貼上錯誤訊息 / log
3. 貼上相關程式碼
4. 說明已試過的方法

重點：要求 AI 先說明根本原因，再給修法
```

**不要讓 AI 直接給你修好的程式碼**，先確認根因分析是對的。

---

## Step 4：評估修法影響範圍

在修之前問自己：
- 這個修法會影響其他功能嗎？
- 有沒有其他地方也有同樣的問題？
- 修完後需要做什麼測試？

---

## Step 5：實作修復

```bash
# 建立 bugfix branch（如果是重要修復）
git checkout -b fix/[bug-描述]

# 實作修復
# ...

# 測試確認 Bug 已修復
# 測試確認其他功能沒有受影響
```

---

## Step 6：防止迴歸

如果專案有測試：
- 新增一個測試，確保這個 Bug 之後不會再出現
- 測試名稱要清楚說明這個 Bug 的情況

---

## Step 7：Commit 並部署

```bash
git add [相關檔案]
git commit -m "fix([scope]): [一句話描述修了什麼]

根本原因：[說明為什麼會發生]
修法：[說明修了什麼]"

git push

# 部署到 production
vercel --prod
```

---

## Step 8：更新記錄

1. `TASKS.md`：把 Bug 從 Bugs 區塊移除，或標記為已修復
2. `CHANGELOG.md`：在 Fixed 區塊加入這個修復
3. 如果 Bug 是因為架構問題引起，在 `docs/DECISIONS.md` 記錄改了什麼、為什麼

---

## P0 緊急處理流程

```
1. 立刻告知相關人員（如果有團隊）
2. 先做快速止血（例如：關掉功能入口、回滾）
3. 找根本原因
4. 實作正確修法
5. 測試後部署
6. 事後分析：為什麼會發生？怎麼防止下次？
```

**P0 不要直接在 main 上修**，開 hotfix branch，測試後再 merge。
