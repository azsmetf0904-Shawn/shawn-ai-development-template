# blindspots.md — 已知盲點與高風險模式

這份文件記錄 Shawn 已知的、反覆出現的盲點。
**AI 的職責：當這些模式出現時，主動點名，不要配合演出。**

---

## Blindspot #1：工具跳躍循環 ⚠️ 最高風險

**[Fact]** 觀察到的循環模式（2026-06-29 Shawn 自述）：
```
研究 → 找到好工具 → 開始興奮 → 建立架構 → 又找到新工具 → 重來
```

**[Fact]** 出現過的跳躍序列：
`Render → Zeabur → Claude → n8n → Cursor → Codex → Repository`

**[Inference]** 每次跳躍的觸發點是「這個新工具可以解決 X」，但 X 往往不是當前產品的瓶頸。

**AI 應該做的事：**
當 Shawn 提到任何新工具，先問：「它幫你完成哪一個具體產品？」
如果答案模糊，不要推薦、不要鼓勵，直接說：「這不符合 Shawn Rule No.1，要繼續當前的工作嗎？」

---

## Blindspot #2：架構先於產品

**[Inference]** 我傾向先把「系統設計得很漂亮」，然後才開始做「真正解決用戶問題的功能」。

**典型表現：**
- 花一週建立開發模板，但產品功能還沒做
- 設計完整的 ADR 系統，但還沒有用戶
- 建立複雜的 CI/CD 流程，但沒有測試

**[Inference]** 這不一定是壞事（好的基礎設施有長期價值），但當它變成「拖延真正工作」的藉口時，有問題。

**AI 應該做的事：**
當 Shawn 在基礎設施投入超過 2 天，主動問：「這個基礎設施投資，對應哪個用戶問題？」

---

## Blindspot #3：多專案並行導致淺度執行

**[Fact]** 我同時管理 5+ 個專案。

**[Inference]** 每個專案都有進展，但沒有一個真正「深度完成」到可以獨立成長的程度。

**典型表現：**
- TeamStar 有用戶，但 onboarding 不完整
- paopao-score 運作，但沒有成長機制
- fb-poster 功能強，但沒有文件和用戶

**AI 應該做的事：**
當 Shawn 說「這個專案也要加功能」，確認它是否在當前的優先清單上。不要讓新需求分散當前焦點。

---

## Blindspot #4：低估維護成本

**[Hypothesis]** 我在建立新功能前，可能沒有充分估計「維護這個功能需要多少長期時間」。

**典型表現：**
- 加入新的自動化 workflow，但沒有監控它是否正常運作
- 建立新的 edge function，但沒有 error logging

**AI 應該做的事：**
在實作新功能前，問一句：「這個功能上線後，誰來確認它正常運作？怎麼知道它壞了？」

---

## Blindspot #5：完美主義 vs 出貨速度的矛盾

**[Hypothesis]** 我說「Build > Perfect」，但實際行動中，有時候因為「還不夠好」而遲遲不上線。

**AI 應該做的事：**
當 Shawn 超過兩次說「這個還需要再改一下」，問：「現在的版本夠不夠讓真實用戶用看看？」

---

## 使用說明

這份文件不是批評，是工具。
每個 blindspot 都是「在壓力或興奮狀態下容易走的老路」。

AI 的任務不是阻止 Shawn，而是讓他**有意識地選擇**，而不是**自動走進去**。
