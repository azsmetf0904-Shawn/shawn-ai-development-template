# Prompt：實作模式

使用時機：要請 Codex / Cursor / Claude Code 實作一個具體功能時。

---

## Prompt 模板（完整版）

```
你現在是這個專案的資深工程師，負責實作以下功能。

## 專案技術棧
[貼入關鍵技術資訊]

## 任務描述
[清楚說明要做什麼]

## 完成標準（Definition of Done）
- [ ] 功能正常運作
- [ ] 邊界情況已處理：[列出邊界情況]
- [ ] 沒有引入安全漏洞
- [ ] 相關 UI 更新完成

## 相關檔案
- [檔案路徑 1]：[說明這個檔案的角色]
- [檔案路徑 2]：[說明這個檔案的角色]

## 限制條件
- 不要改動：[列出不能動的檔案或函式]
- 必須遵守：[列出命名規範、樣式規範等]

## 開始前請先
1. 閱讀並確認你理解以上需求
2. 如果有不清楚的地方，先問我，不要假設
3. 確認後開始實作
```

---

## Prompt 模板（快速版，日常使用）

```
幫我實作 [功能名稱]。

背景：[一句話說清楚]
相關檔案：[路徑]
不能動的東西：[說明]
完成標準：[簡短說明]
```

---

## 使用範例

**場景**：要在任務列表頁面加上篩選功能

```
幫我在任務列表頁面加上「依狀態篩選」功能。

背景：目前 /pages/member/Tasks.js 顯示所有任務，用戶想要只看「待完成」或「已完成」的任務。
相關檔案：src/pages/member/Tasks.js（任務列表），src/lib/supabase.js（fetchTasks 函式）
不能動：fetchTasks 函式的參數介面（其他頁面也在用）
完成標準：頁面頂部有三個按鈕（全部、待完成、已完成），點擊後列表即時更新
```

## 常見錯誤防範

在 prompt 中明確說明以下事項，避免 AI 猜測：

1. **用戶觸發還是自動觸發**：「按鈕點擊觸發」vs「頁面載入自動執行」
2. **資料來源**：「從 Supabase query」vs「從現有 state 過濾」
3. **錯誤處理**：「顯示錯誤訊息」vs「靜默失敗」
4. **loading 狀態**：「需要 loading spinner」vs「不用」
