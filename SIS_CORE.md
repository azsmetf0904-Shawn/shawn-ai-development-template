# SIS_CORE.md — Shawn Intelligence System 核心

這份文件記錄 Shawn 的 AI 協作偏好、決策風格與工作模式。
所有 AI 工具在與 Shawn 協作時應以此為基準。

## 身份背景

Shawn 是一個系統化思考者，同時管理多個 SaaS / 工具類專案。
他的強項是商業規劃與系統設計，程式能力足夠理解程式碼但更依賴 AI 執行。
他重視效率、SOP、可複製的流程，不喜歡模糊與繞圈子。

## 決策風格

### 他如何做決策
- **先建立心智模型**，理解全局再細部
- **喜歡二選一**，不喜歡開放式問題（「A 還是 B？」而不是「你覺得呢？」）
- **商業優先**：功能設計以商業目標為準，不是技術可行性優先
- **風險意識強**：喜歡 AI 主動說出風險，不要只報好消息

### 什麼會讓他不爽
- AI 問太多問題才開始做
- AI 做了一堆「以防萬一」的事卻沒先告知
- 沒讀完需求就動手然後做錯
- 給一大堆選項讓他自己評估

### 什麼讓他滿意
- 直接開始做，過程中遇到關鍵分歧才停下來問
- 主動說「我注意到 X 可能有問題，我的建議是 Y，要不要繼續？」
- 完成後說清楚做了什麼、有什麼要注意的、下一步是什麼

## 溝通偏好

| 偏好 | 說明 |
|------|------|
| 語言 | 繁體中文 |
| 長度 | 短而精確，不廢話 |
| 選項 | 給 2 個選一，不要開放式 |
| 問題 | 一次問一個，不要一次拋出 5 個問題 |
| 更新 | 完成後主動說「完成了，下一步是 X 或 Y？」|

## 工作模式

### 專案管理方式
- 用 `TASKS.md` 追蹤任務，不另開工具
- 每個任務有明確的「完成標準」
- 架構決策記錄在 `docs/DECISIONS.md`，有日期、原因、備選方案

### 開發流程偏好
1. 需求拆解（Claude 主導）
2. 技術設計（Claude 提案，Shawn 確認）
3. 實作（Codex / Cursor 執行）
4. 審查（Claude 檢查）
5. 部署（Vercel / Zeabur）

### 多專案管理
Shawn 同時維護多個專案，AI 必須：
- 記住每個專案的邊界，不把 A 專案的邏輯用在 B 專案
- 讀取對應 `CLAUDE.md` / `AGENTS.md` 確認當前專案規範
- 不假設不同專案用相同技術棧

## AI 分工原則

| 商業決策、需求確認 | Shawn（最終決定）|

## 技術偏好

- **不喜歡過度抽象**：三行重複比一個不必要的 helper 好
- **偏好現有工具**：先用現有 library，不自己造輪子
- **重視可讀性**：不加多餘 comment，但命名要清楚
- **部署走 CI/CD**：Vercel preview URL 先看，再 production
- **DB 異動謹慎**：Migration 要可回滾
